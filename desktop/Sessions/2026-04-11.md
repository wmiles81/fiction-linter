## Decisions made

- **Decision:** Put the convention in a project-scoped `CLAUDE.md` at `fiction-linter/desktop/CLAUDE.md` rather than in a `Sessions/README.md`.
  - **Why:** Claude Code auto-loads `CLAUDE.md` files from the working directory upward, so future sessions pick up the convention without the user having to remind me. A `README.md` would be inert unless someone chose to read it.
  - **Alternatives considered:** Adding the convention to the global `~/CLAUDE.md`. Rejected because `~/CLAUDE.md` is already Arcwright-heavy, and this convention is only relevant under `fiction-linter/desktop/`.

- **Decision:** Session files named `YYYY-MM-DD.md` with `-2`, `-3` suffixes for same-day repeats, and "append to today's file if continuing" as the default.
  - **Why:** Keeps chronological sort trivial, avoids a proliferation of files for short sessions, and makes "what were we doing recently" easy to answer with `ls -t`.
  - **Alternatives considered:** `NNNN-description.md` (sequential). Rejected because it hides the timeline.

- **Decision:** The template's **Verified** section is checklist-style and explicitly honest-only — boxes are only checked for things actually run end-to-end.
  - **Why:** LLMs (including me) have a strong bias toward writing "✅ Implemented X" the moment code typechecks. The checklist format forces a conscious "did I actually run this?" moment before claiming completion.
  - **Alternatives considered:** A free-text "what was verified" field. Rejected because prose is easier to fudge than a checkbox with a command next to it.

- **Decision:** Did not convert the desktop shell from JSX to TSX, did not debounce the lint effect, did not swap the textarea for CodeMirror — even though all three came up in the review.
  - **Why:** The session's job was review + notes infrastructure, not changes to running code. Scope discipline.
  - **How to apply:** These live in the "Next steps" list below, in priority order.

2. **Click-on-issue → cursor-jump wiring.** The `line`/`column` on each finding is computed in [../src/App.jsx](../src/App.jsx#L51-L60) but nothing consumes it. Add an `onIssueClick` prop through [IssueList](../src/components/IssueList.jsx) that moves the textarea selection to the offending range. This also validates the end-to-end finding→position plumbing before the CodeMirror swap.
3. **Swap `<textarea>` for CodeMirror 6** with its `@codemirror/lint` extension. Biggest UX unlock — gets inline squiggles matching the VS Code extension's red/yellow/blue severities. Adapter from `LintFinding[]` to CM6 `Diagnostic[]` is ~30 LOC.

## Gotchas learned

- **VS Code chat history is not in the project tree.** It lives in `~/Library/Application Support/Code/User/workspaceStorage/…`, which is why the user couldn't move their prior session into `Sessions/Session-1`. For future chats where continuity matters, write notes directly into the project with a Claude session file instead.
- **Vite transparently handles TypeScript imports from a JSX entrypoint**, which is why [App.jsx](../src/App.jsx) can `import { PatternLinterCore } from '@shared/linting'` even though the desktop package has no `tsconfig.json`. The alias in [vite.config.ts](../vite.config.ts#L21-L23) resolves to `../src/shared`, and Vite's esbuild transform handles the `.ts` files. Works, but means there's zero type-checking on the consuming side — a TSX migration would fix that.
- **`isInsideQuotes` is scoped to the current line only.** Latent bug in the shared core — quote-crossing-line prose (common) will falsely flag dialogue-only exclusions. Noting here so the next session doesn't rediscover it.
- **Lint runs on every keystroke without debouncing.** Latent perf issue; fine on small files, will stutter on a real manuscript.
- **AI settings are dead plumbing right now.** Four fields flow from UI → preload → main → `userData/settings.json`, but no code actually reads them to call a model. Either build the consumer or delete the plumbing — current state is a footgun where someone will paste an API key into a field that does nothing.
- **`getDefaultSpePath()` in [electron/main.js](../electron/main.js#L7-L9) walks up from `app.getAppPath()`** to find `../resources/spe_defaults`. This works in dev but will break once the app is packaged by electron-builder, because `app.getAppPath()` points into `app.asar`. Worth testing as part of step 5 abov
