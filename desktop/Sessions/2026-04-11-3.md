## Decisions made (with why)

This is the meaningful audit trail. Each entry: the decision, why, and how to apply it later.

### D-1. Single-quoted multi-`-m` commit messages, `git -C /path` prefix, separate Bash calls

Earlier in the session you correctly pushed back when tool calls were getting rejected. Diagnosis: your `~/.claude/settings.json` allow list has very specific patterns like `Bash(git commit -m ':*)` (single-quoted message), and the heredoc commit pattern I used in Phase 0-6 (`git commit -m "$(cat <<'EOF' ... EOF)"`) doesn't match because it starts with a double quote. Combined with what looks like "bypass permissions mode set in UI but not persisted in settings.json," unmatched commands were prompting silently.

- **Why:** working around the allow list patterns is much faster than chasing the bypass mode toggle.
- **How to apply:** every subagent dispatch this session included a "permission constraints" preamble telling the subagent to use `git -C /path`, single-quoted messages, multiple `-m` flags for paragraphs, no `&&` chains, no apostrophes in single-quoted messages (rewrite "doesn't" → "does not"). Worked for all 8 commits without rejection.
- **Permanent fix:** find and toggle bypass-permissions mode in Claude Code (Shift+Tab cycling, or `/permissions` mode toggle) so future sessions don't have to wrestle with this.

### D-2. Plan baseline committed BEFORE Phase 7.1 starts

Same hygiene as Phase 0. The Phase 7 plan + clarifications doc were sitting untracked in the working tree. Committing them as `67148ca` before any code touched lets each phase commit be diff-clean against a real ancestor.

### D-3. Workspace CSS converted from grid to flex (Phase 7.1 deviation)

The Phase 7.1 plan gave `.left-panel` an inline `width` style for the resizer, but the existing `.workspace` rule used `display: grid; grid-template-columns: 300px 1fr;` — and grid templates always win over child inline widths. The implementer caught this and converted to `display: flex` with `.left-panel { flex-shrink: 0 }` and `.right-panel { flex: 1; min-width: 0; min-height: 0 }`.

- **Why:** necessary for the resizer to function. Grid was the wrong layout primitive for a user-resizable panel.
- **How to apply:** when planning resizable panels, always check the existing layout primitive before assuming inline styles will compose.

### D-4. `openFile` re-open replaces content (Phase 7.2 deviation — STILL OUTSTANDING BUG)

The Phase 7.2 plan said `openFile` should switch to an existing tab if the path matches, no content update. But the existing `App.debounce.test.jsx` test simulates editing by clicking the same file twice with different `readFile` stubs — it depends on the store re-loading content on re-open. The implementer modified `openFile` to replace `markdownSource`/`name` and reset `dirty: false` when re-opening.

- **Why:** preserves Phase 0-6 semantics that the debounce test depends on.
- **The cost:** if a user has dirty unsaved edits in a tab and clicks that same file in the FileTree, **their content gets silently overwritten and `dirty` flips false.** This is a real UX bug that gets worse with Phase 7.3's persistent tabs.
- **Correct fix (deferred to a later session):** split into `openFile(path, content)` (new tab or no-op switch) and `revertTab(id)` (explicit reload), with a dirty-check confirm prompt at the App level.
- **How to apply:** see Priority 1 / next-steps section.

### D-5. Zustand selector-per-line pattern, not full destructuring

The Phase 7.2 plan showed `const { settings, setSettings, ... } = useAppStore()` style destructuring. The implementer used `useAppStore(state => state.settings)` per slice. Idiomatic Zustand: avoids re-rendering App on unrelated state changes. Zero behavior change.

### D-6. Phase 7.1 follow-up applied during Phase 7.2

Phase 7.1 review flagged the `localStorage.getItem('fl.leftPanelWidth')` read as un-wrapped (the write side was in try/catch). When Phase 7.2 rewrote App.jsx, the implementer wrapped the read for symmetry. Took zero extra time because App.jsx was already being touched.

- **How to apply:** "Important but not blocking" reviewer findings often have a free fix in the next phase that touches the same file. Worth scanning for these opportunities when dispatching.

### D-7. Phase 7.3 close button aria-label flattened to "Close tab" (deviation)

The plan's `Close tab ${tab.name}` aria-label collided with the tab-name button in jsdom name queries — `getByRole('button', { name: /alpha\.md/ })` would match BOTH the tab-name button AND the close button (because "Close tab alpha.md" contains "alpha.md"). Implementer flattened to `aria-label="Close tab"` and put the per-tab info in the `title` attribute.

- **Cost:** screen-reader users hear only "Close tab" without the file name. Acceptable because the tab name is always adjacent.
- **How to apply:** when adding a new test-targetable element with text that overlaps another, prefer `data-testid` or scope queries to a unique role+name pair from the start.

### D-8. App.debounce.test.jsx scoped to `data-path` (Phase 7.3)

Mounting TabBar caused `getByRole('button', { name: /story\.md/ })` in App.debounce.test.jsx to match BOTH the tree row AND the new tab button. Fixed via `querySelector('.tree-row[data-path="/tmp/story.md"]')`. **This fix made the test substrate-independent** — Phase 7.7's editor swap required ZERO additional changes to this test.

- **The lesson:** when role-based queries become ambiguous after a UI addition, drop to a more specific selector. The Testing Library "always use roles" advice has limits when you add multiple identically-named elements.

### D-9. `.right-panel` grid template extended in Phase 7.3

Was `1fr auto`. Adding TabBar made it `auto 1fr auto`. Phase 7.4 didn't need to extend it again because StatusBar lives at the `app-shell` level (outside `.right-panel`), not inside.

### D-10. EditorPanel updateListener hoisted into useMemo with ref (Phase 7.4 deviation)

The plan inlined `EditorView.updateListener.of(...)` directly in the extensions array literal. The implementer hoisted it into `useMemo` that closes over a ref-stored `onStateChange`.

- **Why:** `@uiw/react-codemirror` rebuilds the entire EditorView when the extensions array identity changes. Inlining the listener would cause a rebuild on every parent render — destroying cursor position, selection, undo history, viewport scroll on every keystroke.
- **The pattern:** stable extension via `useMemo([], ...)`, latest callback via `useRef`. Same pattern as Phase 3's `lintBridgeExtensions` constant. Apply this anywhere a CodeMirror extension needs to call the latest version of a parent callback.

### D-11. StatusBar test count was 13, not 10 (plan was wrong)

The Phase 7.4 plan summary said "10 tests" but the actual test code block had 13 `it()` calls — the plan author (me) miscounted. Implementer just used the real number. Final test count is 77 → not 74 as the plan said.

### D-12. Persist debounce 400ms leading edge (Phase 7.5 bonus, addressing Phase 7.3 review finding)

Phase 7.3 reviewer flagged that `updateContent` (which fires on every keystroke) triggered a sync `fs.writeFileSync` per call. For a 1000-char paragraph that's ~1000 disk writes. Bundled a 400ms leading-edge debounce into Phase 7.5 since that phase already touched `useEditorStore.js`. ~5 lines of code, ~100x reduction in write amplification, no test churn.

### D-13. Persist flush-on-quit (Phase 7.7 bonus, addressing Phase 7.5 review finding)

The 400ms debounce introduced a small regression: dirty UNTITLED tabs (path = null, no disk source to rehydrate from) lose their last 0-400ms of content if the user quits during the debounce window. Persisted-with-path tabs are fine because hydrate re-reads from disk.

- **Fix:** added a `flushPersist()` action to `useEditorStore.js` that clears the timer and writes synchronously. Wired to a `beforeunload` listener in App.jsx.
- **Caveat:** `beforeunload` handlers can't await — they have to start the IPC and hope it completes before the window closes. For Electron's IPC model this should work for small payloads but isn't guaranteed for huge ones. If untitled-tab data loss becomes a real complaint, switch to a `before-quit` IPC from main with a synchronous wait.

### D-14. Custom `styledSpansToSemanticPlugin` for gdoc paste (Phase 7.7 deviation)

The plan's converters.js used a vanilla `rehype-parse → rehype-remark → remark-stringify` pipeline. That pipeline doesn't know about Google Docs' span-with-style approach to formatting (Google Docs renders bold as `<span style="font-weight:700">`, not `<strong>`). The implementer added a small custom rehype plugin that rewrites styled spans to semantic elements BEFORE rehype-remark runs. Without it, gdoc paste tests failed.

- **Why:** real-world gdoc paste needs this to work. Your stated use case from the clarifications doc was "I have copied gdoc directly into md."
- **Plus:** added `rule: '-'`, `ruleRepetition: 3`, `ruleSpaces: false` to remarkStringify options so HRs render as `---` instead of `***`.

### D-15. lintOverlay `ensureBlockSeparator()` helper (Phase 7.7 deviation)

The plan's `buildOffsetMap` used a `lastWasBlock` flag to decide whether to insert `\n\n` before HR. Bug: in `<p>Before</p><hr>`, `lastWasBlock` was already true after the `<p>`, so the helper skipped the separator and the HR landed without proper spacing. Implementer replaced the flag with `ensureBlockSeparator()` that checks `text.endsWith('\n\n')` directly. Matches the test expectations.

### D-16. Editor `lastEmittedRef` guard (Phase 7.7 deviation)

The Editor's `useEffect([value])` converts markdown→HTML and sets `editorRef.current.innerHTML = clean`. Problem: the parent echoes back the markdown the user just typed, which triggers the effect, which re-sets innerHTML, which **wipes the cursor on every keystroke**. Implementer added a `lastEmittedRef` that tracks the last markdown emitted from `handleInput` — when the next `value` matches that ref exactly, the effect skips the innerHTML update.

- **Why:** without this guard, the WYSIWYG editor would be unusable. Cursor would jump to the start of the document on every keystroke.
- **Critical correctness:** the comparison must be exact (not deep). The ref must update on every emission. Test coverage for this in Editor.test.jsx is implicit (the "fires onChange with markdown" test would fail if the cursor were getting wiped, because the input event would never propagate).

### D-17. App.debounce.test.jsx required ZERO modifications (Phase 7.7)

The Phase 7.3 deviation D-8 above made this test fully editor-substrate-independent. After Phase 7.7 swapped CodeMirror for contenteditable, the test ran without any changes. The lint spy on `PatternLinterCore.prototype.lintText` is decoupled from the editor implementation entirely.

- **The dividend:** writing tests against role-based / data-attribute selectors and spying at the boundary between consumer and dependency makes them robust to internal refactors.

## Gotchas learned

- **Permission allow lists are pattern-string-matched, not semantically.** `Bash(git commit -m ':*)` matches single-quoted messages literally. Heredoc patterns (`"$(cat <<'EOF' ... EOF)"`) start with `"` and don't match. The fix wasn't to dig through Claude Code's permission-mode runtime state — it was to match the existing pattern with `git -C /path commit -m 'subject' -m 'body para 1' -m 'body para 2'`. **Workaround beats spelunking.** Every subagent dispatch this session included an explicit "permission constraints" section that taught the subagent the same workaround. Worked first try in 7 of 7 implementations and 7 of 7 reviews.

- **Plan task ordering matters for TDD discipline.** The Phase 7.6 plan listed Task 7.6.1 (modelCatalog impl) BEFORE Task 7.6.2 (modelCatalog test). The implementer followed the plan order, so the test was green on first run instead of going through a proper red phase. The plan author (me) should always order tests before implementations. The correctness wasn't compromised but the TDD signal was lost.

- **`@uiw/react-codemirror` rebuilds the entire EditorView when extension array identity changes.** Inlining `EditorView.updateListener.of(...)` in the JSX rebuilds on every render and destroys cursor state. The fix is `useMemo([], () => listener)` plus a `useRef` for the latest callback. Same pattern showed up twice this session: Phase 3's `lintBridgeExtensions` constant and Phase 7.4's hoisted updateListener. **Any object identity passed to a CodeMirror extension prop must be stable across renders.**

- **contenteditable + React requires `lastEmittedRef` to prevent cursor wipes.** The pattern: parent stores markdown, child renders HTML in contenteditable, child emits markdown on input, parent echoes markdown back as a prop. The naive `useEffect([value])` that re-sets innerHTML will wipe the cursor on every keystroke because the parent's echo IS the value the child just emitted. The guard: track the last-emitted value in a ref, skip the effect when the new value matches.

- **CSS Custom Highlight API has no jsdom polyfill.** Tests can't verify highlights actually render. The fallback path in `applyLintHighlights` is `if (typeof CSS === 'undefined' || !CSS.highlights) return () => {}` — a no-op that lets the unit tests pass but provides zero signal about real-Electron behavior. **The first `npm run dev` is the only validation.**

- **Google Docs renders bold/italic/underline as `<span style="...">`, not as semantic `<strong>` / `<em>` / `<u>`.** Vanilla `rehype-remark` doesn't pick up styled spans because it expects semantic HTML. A small custom rehype plugin (15 lines) that rewrites styled spans BEFORE rehype-remark runs handles this cleanly.

- **Vitest 4.x rejects `require('vitest')` in test files.** The error message is explicit: `Vitest cannot be imported in a CommonJS module using require(). ## Phase 7.7 review addendum

The Phase 7.7 review landed during session-note drafting. **✅ Approved**, but flagged one real bug worth fixing promptly and several refinements:

### Important — fix promptly

**A. `isSettingContentRef` cleanup gap in `editor/Editor.jsx` (silent input drop bug).** The value-sync effect sets `isSettingContentRef.current = true` BEFORE `markdownToHtml(value)` and unsets it inside the resolved `.then`. The cancellation cleanup uses a `cancelled` flag that early-returns from the `.then` BEFORE setting the ref back to false. **A rapid `value` change can leave the ref stuck `true` during the overlap window — silently dropping a real `input` event.** This is a real bug that won't show up in tests because it requires specific async timing.

**Fix (one line):** add `.finally(() => { isSettingContentRef.current = false; })` to the promise chain in the value-sync effect.

**B. Async race on `handleInput`.** Multiple in-flight `htmlToMarkdown(html).then(...)` calls can resolve out of order if the user types fast. The later-initiated call may resolve first, so `onChange` fires with stale markdown and `lastEmittedRef` records the wrong value. Real risk in real use, no defense currently. Fix: AbortController or sequence number on each call.

**C. `flushPersist` `beforeunload` reliability is unverified.** The handler dispatches the IPC synchronously but `tabs:save` runs in main as `async writeTabs(state)`. If main uses `fs.promises.writeFile` (async) the write may not complete before the renderer is killed on hard quit. Worth verifying in real Electron and switching to `fs.writeFileSync` if needed.

**D. `handleInput` double-fires after toolbar commands.** `handleToolbarCommand` calls `handleInput()` after `execCommand`, but `execCommand` already fires a native `input` event — so two `htmlToMarkdown` conversions run on the same content. Wasteful, compounds bug B. Fix: remove the explicit `handleInput()` call.

### Minor refinements (Phase 8 cleanup territory)

- **Dead code:** `findEntryContaining` filters `isSynthetic` entries but `buildOffsetMap` never pushes any. Either delete the dead branch or actually push synthetic entries so lookups inside `---` blocks have somewhere to land.
- **Triple-pass paste pipeline** (htmlToMarkdown → markdownToHtml → sanitizeHtml) is defense-in-depth but expensive. Tens of ms for paragraph paste, possibly seconds for chapter-sized paste. Worth a perf budget test.
- **Toolbar `canUndo/canRedo/canSave={true}` are hardcoded placeholders** in Editor.jsx. Save is always enabled even when not dirty; Undo/Redo always enabled even with empty history. Wire to real state in Phase 8.
